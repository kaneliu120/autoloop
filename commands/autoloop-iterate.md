# AutoLoop T3: Iterate — 目标驱动迭代

## 执行前提

读取 `autoloop-plan.md` 获取：
- KPI 定义（必须是可测量的，有具体数值）
- 当前基线（第一轮开始前测量）
- 改动范围约束（只改什么，不改什么）
- 测量方法（如何在每轮后重新测量 KPI）
- 最大轮次

如果 KPI 定义不清晰（"质量更好"而不是"错误率 < 0.5%"），先帮用户将其量化。

---

## KPI 量化规则

模糊 KPI → 具体 KPI 的转换示例：

| 模糊描述 | 量化 KPI |
|---------|---------|
| "更快" | API P95 延迟 < 200ms |
| "更好的质量" | 测试覆盖率 ≥ 80% + py_compile 零错误 |
| "减少错误" | 异常日志数量下降 > 80% |
| "代码更清晰" | 函数平均行数 < 30 + 无 TODO 注释 |
| "更好的用户体验" | 首屏时间 < 1.5s + Lighthouse 分数 ≥ 90 |

---

## 第 0 步：基线测量（只执行一次）

在任何改动之前，测量当前状态：

```
基线测量报告：

KPI 指标：{指标名}
测量方法：{具体命令或方法}
当前值：{测量结果}
目标值：{目标}
差距：{差距量，以百分比或绝对值表示}
达标难度估计：{低/中/高}（基于差距和已知改进空间）
```

写入 autoloop-progress.md 的"基线"部分。

---

## 每轮执行流程

### OBSERVE：差距分析

```
当前 KPI：{值} / 目标：{值} / 差距：{量}
已消耗轮次：{N} / {最大}
本轮可用预算：充裕 / 有限 / 告急

关键观察：
- {本轮最重要的限制因素是什么}
- {上轮改进的效果如何}
- {是否有意外发现}
```

### ORIENT：改进策略制定

根据差距类型选择策略：

**差距 > 50%（还很远）**：
- 策略：寻找"大动作"（架构改变、算法替换、根本性重构）
- 不要在小优化上浪费时间

**差距 20-50%（有进展但不够）**：
- 策略：组合多个中等改进（逐一优化热点路径）
- 用 profiling/分析工具找到瓶颈

**差距 < 20%（接近目标）**：
- 策略：精细调优（参数调整、缓存策略、并发调整）
- 注意收益递减

**连续 2 轮改进 < 5%**：
- 当前方法已到极限，必须换思路
- 考虑：换算法、改架构、重新定义 KPI（是否测量了正确的东西）

### DECIDE：本轮行动计划

```
本轮行动计划：
1. {具体行动 1}（预期改进：{预估值}）
   执行者：{agent 类型}
   文件：{绝对路径}

2. {具体行动 2}（预期改进：{预估值}）
   执行者：{agent 类型}
   文件：{绝对路径}

独立性分析：
- 行动 1 和行动 2 独立？{是/否}
- 如果否，执行顺序：{1 先 / 2 先}
```

### ACT：执行改进

分配 subagent 执行改进。每个 subagent 必须收到：

```
你是 {类型} subagent，负责以下改进任务：

目标 KPI：{KPI 名称} 从 {当前值} 降低到 {目标值}

本次改进任务：
{具体任务描述}

改动约束：
- 可以修改：{允许修改的范围}
- 不可修改：{禁止修改的内容}
- 保持不变：{API 接口 / 数据库 schema / 其他约束}

相关文件（绝对路径）：
- {文件 1}
- {文件 2}

执行后验证：
- 运行：{验证命令}
- 期望结果：{预期输出}

输出：
1. 修改了哪些文件，每个文件改了什么
2. 验证结果截图/输出
3. 预期对 KPI 的影响
4. 是否引入了新的问题
```

### VERIFY：重新测量

所有 subagent 完成后，重新运行测量：

```
改进后 KPI 测量：

测量命令：{命令}
结果：{值}

对比：
  改进前：{基线值}
  本轮后：{当前值}
  本轮改进：{改进量}（{改进比例}%）
  累计改进：{总改进量}

是否达标：{是/否}
  如果否，剩余差距：{量}
```

## 人工反馈介入点

在以下情况主动暂停，请求用户反馈：

**情况 1：KPI 改进但体验变差**
```
检测到潜在问题：
  KPI {指标} 从 {X} 改善到 {Y}（↑）
  但 {其他指标} 从 {X} 降低到 {Y}（↓）

  这是可接受的权衡吗？继续 / 回滚 / 调整目标？
```

**情况 2：发现根本性问题**
```
发现根本性问题，需要决策：

  问题：{描述}
  影响：{影响范围}

  选项：
  A. 处理根本问题（预计需要 {N} 轮，影响 {范围}）
  B. 绕过问题继续（有风险：{风险描述}）
  C. 重新定义 KPI 目标

  你选择？
```

**情况 3：连续无进展**
```
连续 {N} 轮改进幅度 < 5%：

  当前 KPI：{值}（目标：{值}）

  可能原因：
  1. {可能原因 1}
  2. {可能原因 2}

  建议：
  A. 换策略（{具体新策略}）
  B. 放弃该方向，尝试 {替代路径}
  C. 接受当前结果（距离目标 {X}%）

  你选择？
```

## 回滚机制

每次改动前在进度文件中记录回滚点：

```markdown
### 回滚点 {N}（第 {M} 轮开始前）

状态快照：
- KPI 值：{值}
- 已修改文件：
  - {文件 1}（git commit: {hash}）
  - {文件 2}（git commit: {hash}）

回滚命令（含安全检查）：
# 步骤 1：确认无非本轮的未提交修改
git status

# 步骤 2：如有脏文件，先暂存
git stash  # 仅在 git status 显示非预期文件时执行

# 步骤 3：用 git revert 回滚（保留完整历史，不使用 git checkout）
git revert {commit_hash}

# 步骤 4：恢复暂存的工作（如步骤 2 执行了 stash）
git stash pop
```

**回滚安全规则**：
- 必须先 `git status` 确认无非本轮的未提交修改
- 有脏文件时先 `git stash` 保存，回滚后 `git stash pop` 恢复
- 使用 `git revert {commit}` 而非 `git checkout -- {file}`，保留完整历史
- 不使用 `git reset --hard`（会销毁未提交的工作）

```markdown
## 第 {N} 轮 — {时间}

**KPI 趋势**：
  基线 → 第1轮 → ... → 本轮
  {基} → {v1} → ... → {vN}

**本轮行动**：
  - {行动 1}：{结果}
  - {行动 2}：{结果}

**改进量**：{上轮} → {本轮}（{+/-X}，{X}%改进）
**累计改进**：{基线} → {本轮}（{X}%改进）

**下一轮策略**：{策略 | 已达标，终止}
```

- 最终值：{值}
- 总改进：{改进量}（{百分比}%）

- 目标达成：{是/否（{当前值} vs 目标 {目标值}）}

## 改进轨迹

| 轮次 | KPI 值 | 改进量 | 主要行动 |
|------|--------|--------|---------|
| 基线 | {值} | — | — |

| 第 N 轮 | {值} | {量} | {行动摘要} |

## 关键改进

1. {最有效的改进}：贡献 {X}% 的总改进量

- {方向 1}：原因 {分析}
