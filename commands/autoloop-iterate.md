# AutoLoop T3: Iterate — 目标驱动迭代

## 执行前提

读取 `autoloop-plan.md` 获取：
- KPI 定义（必须是可测量的，有具体数值）
- 当前基线（第一轮开始前测量）
- 改动范围约束（只改什么，不改什么）
- 测量方法（如何在每轮后重新测量 KPI）
- 最大轮次

如果 KPI 定义不清晰（"质量更好"而不是"错误率 < 0.5%"），先帮用户将其量化。

**Round 2+ OBSERVE 起点**：先读取 `autoloop-findings.md` 反思章节，获取遗留问题、有效/无效策略、已识别模式、经验教训，再扫描当前状态。详见 `protocols/loop-protocol.md` OBSERVE Step 0 章节。

## KPI 量化规则

> 以下为非规范性示例，实际 KPI 应根据任务目标动态量化。

| 模糊描述 | 量化 KPI |
|---------|---------|
| "更快" | API P95 延迟 < 200ms |
| "更好的质量" | 测试覆盖率 ≥ 80% + py_compile 零错误 |
| "减少错误" | 异常日志数量下降 > 80% |
| "代码更清晰" | 函数平均行数 < 30 + 无 TODO 注释 |
| "更好的用户体验" | 首屏时间 < 1.5s + Lighthouse 分数 ≥ 90 |

## 每轮执行流程

### OBSERVE：差距分析

```text
读取 autoloop-findings.md 的反思章节（Round 2+ 必执行）
获取遗留问题、有效/无效策略、已识别模式、经验教训
（Step 0 规范见 protocols/loop-protocol.md OBSERVE Step 0 章节）

当前 KPI：{值} / 目标：{值} / 差距：{量}
已消耗轮次：{N} / {最大}
本轮可用预算：充裕 / 有限 / 告急

关键观察：
- {本轮最重要的限制因素是什么}
- {上轮改进的效果如何}
- {是否有意外发现}
```

### ORIENT：改进策略制定

根据差距类型选择策略：

**差距 > 50%（还很远）**：
- 策略：寻找"大动作"（架构改变、算法替换、根本性重构）
- 不要在小优化上浪费时间

**差距 20-50%（有进展但不够）**：
- 策略：组合多个中等改进（逐一优化热点路径）
- 用 profiling/分析工具找到瓶颈

**差距 < 20%（接近目标）**：
- 策略：精细调优（参数调整、缓存策略、并发调整）
- 注意收益递减

**连续无进展**（阈值见 `protocols/evolution-rules.md` 进化类型 3 策略切换触发条件）：
- 当前方法已到极限，必须换思路
- 考虑：换算法、改架构、重新定义 KPI（是否测量了正确的东西）

### DECIDE：本轮行动计划

```text
本轮行动计划：
1. {具体行动 1}（预期改进：{预估值}）
   执行者：{agent 类型}
   文件：{绝对路径}

2. {具体行动 2}（预期改进：{预估值}）
   执行者：{agent 类型}
   文件：{绝对路径}

独立性分析：
- 行动 1 和行动 2 独立？{是/否}
- 如果否，执行顺序：{1 先 / 2 先}
```

并行/串行判断规则见 `protocols/agent-dispatch.md`。

### ACT：执行改进

#### 派遣

角色：按任务类型选择（backend-dev / frontend-dev / fixer 等，职责定义见 `protocols/agent-dispatch.md`）

#### 本次范围

- 目标 KPI：{KPI 名称} 从 {当前值} 到 {目标值}
- 改进任务：{具体任务描述}
- 可修改范围：{允许修改的范围}
- 不可修改：{禁止修改的内容}
- 相关文件：{绝对路径列表}

#### 执行流程

1. 读取对应 protocol 获取规范
2. 根据任务描述制定实施方案
3. 实施修改，每步验证
4. 输出：修改文件列表 + 验证结果 + 预期 KPI 影响 + 是否引入新问题

### VERIFY：重新测量

KPI 门禁定义见 `protocols/quality-gates.md` T3 KPI Target 章节。

```text
改进后 KPI 测量：

测量命令：{命令}
结果：{值}

对比：
  改进前：{基线值}
  本轮后：{当前值}
  本轮改进：{改进量}（{改进比例}%）
  累计改进：{总改进量}

是否达标：{是/否}
  如果否，剩余差距：{量}
```

## 回滚机制

每次改动前在进度文件中记录回滚点：

```markdown
### 回滚点 {N}（第 {M} 轮开始前）

状态快照：
- KPI 值：{值}
- 已修改文件：
  - {文件 1}（git commit: {hash}）
  - {文件 2}（git commit: {hash}）

回滚命令（含安全检查）：
# 步骤 1：确认无非本轮的未提交修改
git status

# 步骤 2：如有脏文件，先暂存
git stash  # 仅在 git status 显示非预期文件时执行

# 步骤 3：用 git revert 回滚（保留完整历史）
git revert {commit_hash}

# 步骤 4：恢复暂存的工作（如步骤 2 执行了 stash）
git stash pop
```

**回滚安全规则**：
- 必须先 `git status` 确认无非本轮的未提交修改
- 有脏文件时先 `git stash` 保存，回滚后 `git stash pop` 恢复
- 使用 `git revert {commit}` 而非 `git checkout -- {file}`，保留完整历史
- 不使用 `git reset --hard`（会销毁未提交的工作）

```markdown
## 第 {N} 轮 — {时间}

  基线 → 第1轮 → ... → 本轮
  {基} → {v1} → ... → {vN}

**本轮行动**：
  - {行动 1}：{结果}
  - {行动 2}：{结果}

**改进量**：{上轮} → {本轮}（{+/-X}，{X}%改进）
**累计改进**：{基线} → {本轮}（{X}%改进）

**下一轮策略**：{策略 | 已达标，终止}
```

## 每轮 REFLECT 执行规范

每轮 EVOLVE 判断之后，在进入下一轮 OBSERVE 之前执行。REFLECT 必须写入文件，不能只在思考中完成（规范见 `protocols/loop-protocol.md` REFLECT 章节）：

写入 `autoloop-findings.md` 的4层反思结构表（问题登记/策略复盘/模式识别/经验教训），格式见 `templates/findings-template.md`：

- **问题登记**：记录本轮发现的 KPI 偏差、改进副作用、测量误差
- **策略复盘**：本轮改进策略的效果评估 — 实际改进量 vs 预期，保持 | 避免 | 待验证（策略评价枚举见 protocols/loop-protocol.md 统一状态枚举）
- **模式识别**：KPI 改进轨迹趋势（线性/指数/收益递减）、哪类改进最有效
- **经验教训**：本轮对目标系统的新认知、哪些假设被验证或推翻
