# AutoLoop 自动化断点分析与控制器方案

**日期**: 2026-03-28
**依据**: T3 迭代 R11 (5.8/10) + 全流程执行追踪
**核心诊断**: AutoLoop 当前是"操作手册"（protocol），不是"自动化引擎"（engine）

---

## 核心问题

AutoLoop 声称是"自主迭代引擎"，但实际执行中**每个阶段之间的推进都依赖 LLM 自觉读取 markdown 并"决定"做下一步**。没有循环控制器驱动 OODA 8 阶段自动推进。

40 个断点分为 3 层：

---

## 第一层：循环引擎不存在（5 个关键断点）

这一层是最根本的问题 — **没有自动推进机制**，所有阶段间的衔接都是"LLM 应该知道要做什么"。

### B1: 入口无模式选择

**位置**: `/autoloop` → `autoloop.md` → `autoloop-plan.md`

**问题**: 用户调用 `/autoloop` 后，系统不自动决定使用 JSON SSOT 模式还是 markdown flat-file 模式。
- `autoloop-init.py` 创建 4 个 markdown 文件
- `autoloop-state.py init` 创建 `autoloop-state.json`
- **两者都不会被自动调用**，取决于 LLM 读到哪段协议

**影响**: 从第一步开始就分裂为两条不确定的路径。

**修复方向**: 控制器启动时根据配置自动选择模式，SSOT JSON 为默认。

---

### B4: VERIFY 无自动评分

**位置**: ACT 阶段完成后 → VERIFY 阶段

**问题**: `autoloop-score.py` 存在且支持 T1-T7 多模板评分，但：
- 没有 hook 自动调用它
- 没有命令流程要求调用它
- LLM 必须"记得"要调用评分器或委派评分 subagent

**影响**: VERIFY 阶段的评分可能被跳过或手动估算，门禁判定不可靠。

**修复方向**: 控制器在 ACT 完成后自动调用 `autoloop-score.py <work_dir>`，将结果注入 VERIFY。

---

### B8: EVOLVE→REFLECT→OBSERVE 无自动推进

**位置**: 每轮结束 → 下一轮开始

**问题**: REFLECT 写入 findings.md 后，系统进入"等待状态" — 没有机制自动：
1. 判断是否继续（EVOLVE 的终止判断结果需要被消费）
2. 推进到下一轮（调用 `add-iteration`）
3. 开始下一轮 OBSERVE（读取上轮 REFLECT 输出）

**影响**: 多轮迭代依赖 LLM 主动记住"现在该做什么"，在上下文压缩后容易丢失。

**修复方向**: 控制器持有轮次计数器，REFLECT 完成后自动评估终止条件，不满足则自动推进。

---

### B13: 无断点恢复

**位置**: 会话中断（上下文耗尽、用户关闭、网络断开）

**问题**: 如果会话在循环中间结束：
- Markdown 模式：4 个文件存在但无 checkpoint 标记，LLM 不知道在哪个阶段中断
- SSOT 模式：JSON 有 phase 字段但无 resume 协议
- 两种模式都没有"从第 N 轮 X 阶段恢复"的自动化机制

**影响**: 每次中断后需要人工告诉 LLM 当前状态，浪费时间且容易出错。

**修复方向**: 控制器维护 `checkpoint.json`（current_round, current_phase, last_completed_phase, timestamp），新会话自动读取并恢复。

---

### B16: Plan→执行无自动连接

**位置**: `autoloop-plan.md` 创建完成 → 第一轮执行

**问题**: plan 创建后，`autoloop.md` line 130 说"进入第一轮执行"，但：
- 没有代码自动调用对应模板命令（如 `/autoloop:research`）
- plan 向导说"5 秒后自动开始"但这是给 LLM 的提示，不是计时器
- Bootstrap 文件创建和执行开始之间没有确定性连接

**影响**: plan 创建后可能停在那里等用户手动触发。

**修复方向**: 控制器在 bootstrap 完成后自动进入 Round 1 OBSERVE。

---

## 第二层：工具链断裂（7 个断点）

脚本工具都存在，但它们是"被动工具" — 需要有人调用。没有"主动编排者"来按顺序调用它们。

### B2: Bootstrap 不读模板

**位置**: `autoloop-init.py` 创建 4 个文件

**问题**: init.py 创建的是空骨架（最少内容），不引用：
- `findings-template.md`（完整的问题追踪表、策略评估表、4 层反思结构）
- `progress-template.md`（8 阶段循环记录模板）
- `quality-gates.md`（模板特定门禁阈值）

**影响**: 创建的文件缺少结构，LLM 需要另外读模板再手动填充。

**修复方向**: 控制器调用增强版 init，自动读取模板和门禁定义，生成结构完整的初始文件。

---

### B3: SSOT 模式无自动 render

**位置**: `autoloop-state.json` 存在时

**问题**: 如果用 SSOT 模式（state.json 为主），需要 `autoloop-render.py` 生成 4 个 markdown 视图文件供 LLM 阅读。但：
- 没有机制在 state.json 更新后自动调用 render
- 如果只更新 JSON 不 render，markdown 和 JSON 不同步
- 没有"哪个是权威"的自动判断

**影响**: JSON 和 markdown 漂移，跨文件验证失败。

**修复方向**: 控制器在每次 state.json 写入后自动调用 render.py，markdown 始终是 JSON 的派生视图。

---

### B9: 经验库无写入

**位置**: REFLECT 阶段 → `experience-registry.md`

**问题**: 7 个命令文件都声明"经验写回: 将本轮策略效果写入 experience-registry.md"，但：
- 这只是 markdown 文档中的一行文字指令
- 没有脚本工具执行写入
- 没有 hook 在 REFLECT 写入后自动追加
- experience-registry.md 的策略效果表当前为空（"初始为空，随任务执行积累"）

**影响**: 跨任务学习完全不存在。自进化维度的分数天花板。

**修复方向**: 控制器在 REFLECT 阶段自动提取策略效果数据，调用专用工具写入 experience-registry。

---

### B10: 经验库无读取

**位置**: OBSERVE Step 0 → `experience-registry.md`

**问题**: loop-protocol.md 声明"还须读取 experience-registry.md"，7 个命令文件都有"经验库读取"步骤，但：
- 没有脚本解析 experience-registry.md 并输出推荐策略
- 没有结构化的"推荐策略列表"传递给 DECIDE
- LLM 需要自己打开文件、找到相关条目、判断是否适用

**影响**: 即使经验库有数据（B9 修复后），也无法被自动消费。

**修复方向**: 控制器在 OBSERVE 阶段自动查询经验库，按 template + context_tags 过滤，输出 recommended_strategies.json。

---

### B19: 门禁阈值不自动填入 plan

**位置**: `autoloop-init.py` 创建 `autoloop-plan.md`

**问题**: plan.md 的质量门禁表创建时是空的（"—" 占位符），但 quality-gates.md 已经定义了每个模板的所有门禁阈值。init.py 不读取 quality-gates.md 来自动填充。

**影响**: LLM 需要手动读 quality-gates.md 再填入 plan，容易遗漏或填错。

**修复方向**: 控制器在 bootstrap 时根据模板类型自动从 score.py 的门禁定义中提取阈值，注入 plan。

---

### B34: 最终报告无自动生成

**位置**: 循环终止后

**问题**: `report-template.md` 和 `audit-template.md` 定义了最终报告格式，但：
- 没有 `autoloop-finalize.py` 工具来填充模板
- 没有机制在 EVOLVE 判定终止后自动生成报告
- LLM 需要手动读模板、读 findings/progress/results、拼接报告

**影响**: 终止后的产出物依赖 LLM 手动整合，质量不稳定。

**修复方向**: 控制器在终止时自动调用 finalize 工具，从 state.json 生成完整报告。

---

### B35: TSV 行不自动追加

**位置**: ACT/VERIFY 阶段 → `autoloop-results.tsv`

**问题**: `autoloop-state.py add-tsv-row` 存在，但：
- 没有命令文件调用它
- VERIFY 评分后不自动将结果写入 TSV
- TSV 在 bootstrap 时只有表头，之后从不追加

**影响**: results.tsv 始终为空或不完整，结构化迭代日志不可用。

**修复方向**: 控制器在 VERIFY 评分完成后自动构造 TSV 行并追加。

---

## 第三层：验证和护栏缺失（5 个关键断点）

协议声明了规则，但没有代码强制执行。

### B11: DECIDE 不强制策略选择规则

**位置**: DECIDE 阶段选择策略

**问题**: loop-protocol.md 声明：
- "避免"策略不应被重复使用
- "保持"策略应优先选择
- 但 DECIDE 阶段没有任何验证机制检查策略历史

**影响**: 同一个失败策略可能被反复尝试，浪费轮次。

**修复方向**: 控制器在 DECIDE 前读取历史策略评价，自动过滤"避免"、提升"保持"。

---

### B26: 振荡检测未实现

**位置**: EVOLVE 阶段终止判断

**问题**: loop-protocol.md line 362 定义了振荡检测（同一维度分数 ±0.5 连续 3 轮），但：
- 没有代码实现此检测
- EVOLVE 决策树不包含振荡检查分支
- `autoloop-variance.py` 存在但仅处理单轮方差，不做跨轮振荡检测

**影响**: 系统可能在两个策略间来回切换而无法收敛。

**修复方向**: 控制器在 EVOLVE 阶段自动计算最近 N 轮的维度分数趋势，检测振荡模式。

---

### B28: side_effect 未验证

**位置**: VERIFY 阶段 → results.tsv side_effect 列

**问题**: loop-protocol.md 声明"不允许未经验证直接填'无'"，但：
- results.tsv 的 side_effect 列接受任何字符串
- 没有验证器检查是否真的运行了受影响维度的评估
- LLM 可以直接写"无"跳过副作用检查

**影响**: 策略的副作用被忽略，可能在一个维度改善的同时恶化其他维度。

**修复方向**: 控制器在 VERIFY 阶段对所有受影响维度运行门禁评估，side_effect 由评估结果自动填充。

---

### B32: 最大轮次未强制

**位置**: EVOLVE 阶段终止判断

**问题**: plan.md 定义了 max_rounds，但：
- 没有代码在 EVOLVE 阶段检查当前轮次是否达到上限
- LLM 可能忽略预算限制继续执行
- `autoloop-state.py` 有 budget.max_rounds 字段但无强制检查

**影响**: 可能无限循环消耗资源。

**修复方向**: 控制器在每轮 EVOLVE 前检查 current_round >= max_rounds，达到则强制终止。

---

### B37: 暂停确认未实现

**位置**: Hard Gate 失败 → 应暂停等待用户确认

**问题**: quality-gates.md 定义了 Hard Gate 失败应暂停（如 T5 Phase 0.5、Phase 5），但：
- 没有代码在 Hard Gate 失败时创建暂停检查点
- 没有"等待用户输入"的机制
- LLM 可能直接跳过暂停继续执行

**影响**: 需要人工决策的关键节点可能被自动跳过。

**修复方向**: 控制器在 Hard Gate 失败时写入 `pause_checkpoint.json`，等待用户确认信号才继续。

---

## 控制器方案：autoloop-controller.py

### 架构

```
autoloop-controller.py <work_dir> [--resume]
  │
  ├── 1. 初始化
  │   ├── 读取 autoloop-plan.md 或 autoloop-state.json
  │   ├── 确定模板类型（T1-T7）
  │   ├── 选择模式（SSOT JSON 为默认）
  │   ├── 如果 --resume：读取 checkpoint.json 恢复状态
  │   └── 创建/验证 bootstrap 文件（自动填充模板+门禁阈值）
  │
  ├── 2. 主循环
  │   ├── OBSERVE
  │   │   ├── 读取 state.json 当前状态
  │   │   ├── 查询 experience-registry（按 template + context_tags 过滤）
  │   │   ├── 读取上轮 REFLECT 输出（Round 2+）
  │   │   └── 输出 observation.json
  │   │
  │   ├── ORIENT
  │   │   ├── 计算差距（当前分数 vs 目标）
  │   │   ├── 分类差距类型（>50% / 20-50% / <20%）
  │   │   └── 输出 analysis.json
  │   │
  │   ├── DECIDE
  │   │   ├── 读取 recommended_strategies（从 OBSERVE）
  │   │   ├── 过滤历史"避免"策略，优先"保持"策略
  │   │   ├── 执行 impact_analysis（受影响维度列表）
  │   │   ├── 选择单一策略（strategy_id 唯一）
  │   │   └── 输出 strategy.json + action_plan.json
  │   │
  │   ├── ACT
  │   │   ├── 按 agent-dispatch.md 生成工单（带完整 context schema）
  │   │   ├── 委派 subagent 执行（独立任务并行，依赖任务串行）
  │   │   ├── 收集 subagent 输出
  │   │   └── 输出 act_results.json
  │   │
  │   ├── VERIFY
  │   │   ├── 自动调用 autoloop-score.py <work_dir>（模板特定门禁）
  │   │   ├── 检查所有受影响维度（side_effect 验证）
  │   │   ├── 调用 autoloop-variance.py（置信度）
  │   │   ├── 构造 TSV 行并追加
  │   │   └── 输出 gate_status.json
  │   │
  │   ├── SYNTHESIZE
  │   │   ├── 合并所有 subagent 输出
  │   │   ├── 运行矛盾检测
  │   │   ├── 生成综合报告
  │   │   └── 输出 synthesis.json
  │   │
  │   ├── EVOLVE
  │   │   ├── 检查终止条件：
  │   │   │   ├── 所有 Hard Gate 通过？→ 正常终止
  │   │   │   ├── current_round >= max_rounds？→ 预算终止
  │   │   │   ├── 振荡检测（最近 3 轮 ±阈值）？→ 策略切换
  │   │   │   ├── 停滞检测（delta < 模板阈值）？→ 策略切换
  │   │   │   └── Hard Gate 失败需用户确认？→ 暂停
  │   │   └── 输出 evolve_decision.json（continue/stop/pause/switch）
  │   │
  │   ├── REFLECT
  │   │   ├── 写入 findings.md 4 层反思结构
  │   │   ├── 写入 experience-registry（策略效果数据）
  │   │   ├── 更新 state.json
  │   │   ├── 调用 render.py 同步 markdown 视图
  │   │   └── 更新 checkpoint.json
  │   │
  │   └── 循环判断
  │       ├── evolve_decision == "continue" → 下一轮 OBSERVE
  │       ├── evolve_decision == "stop" → 退出循环
  │       ├── evolve_decision == "pause" → 写 pause_checkpoint，等待用户
  │       └── evolve_decision == "switch" → 更新策略，下一轮 OBSERVE
  │
  └── 3. 终止
      ├── 调用 autoloop-finalize.py 生成最终报告
      ├── 更新 state.json 终止状态
      ├── 调用 autoloop-validate.py 最终一致性检查
      └── 输出终止摘要

```

### 数据流

```
                    experience-registry.md
                           ↑ write (REFLECT)
                           ↓ read (OBSERVE)

plan.md ──→ controller ──→ state.json ──→ render.py ──→ 4个markdown视图
                │                ↑
                │                │ update
                ↓                │
          subagents ──→ act_results ──→ score.py ──→ gate_status
                                           │
                                           ↓
                                     results.tsv (自动追加)
```

### checkpoint.json 结构

```json
{
  "task_id": "T1-RESEARCH-001",
  "current_round": 3,
  "current_phase": "VERIFY",
  "last_completed_phase": "ACT",
  "timestamp": "2026-03-28T15:30:00Z",
  "evolve_history": [
    {"round": 1, "decision": "continue", "reason": "差距>50%"},
    {"round": 2, "decision": "continue", "reason": "delta +1.2"}
  ],
  "pause_state": null
}
```

### 与现有工具的集成

| 现有工具 | 控制器如何调用 | 调用时机 |
|---------|------------|---------|
| autoloop-init.py | 初始化阶段 | Bootstrap（增强版，读模板+门禁） |
| autoloop-state.py | 每个阶段 | 读/写 state.json |
| autoloop-render.py | REFLECT 后 | 同步 markdown 视图 |
| autoloop-score.py | VERIFY 阶段 | 自动评分 |
| autoloop-validate.py | 终止时 | 最终一致性检查 |
| autoloop-variance.py | VERIFY 阶段 | 置信度计算 |
| autoloop-finalize.py | 终止时 | 生成最终报告（新建） |

### 需要新建的工具

| 工具 | 职责 |
|------|------|
| `autoloop-controller.py` | 主循环控制器（本方案核心） |
| `autoloop-finalize.py` | 终止时生成最终报告 |
| `autoloop-experience.py` | 经验库读写工具（从 REFLECT 提取+查询推荐） |

### 实施阶段

| 阶段 | 内容 | 解决断点 |
|------|------|---------|
| Phase 1 | controller.py 骨架 + checkpoint + 自动轮次推进 | B1, B8, B13, B16 |
| Phase 2 | VERIFY 自动评分 + TSV 追加 + render 同步 | B4, B3, B35 |
| Phase 3 | 经验库读写工具 + OBSERVE/REFLECT 集成 | B9, B10 |
| Phase 4 | 护栏（振荡检测+max_rounds+暂停确认+side_effect）| B26, B32, B37, B28 |
| Phase 5 | Bootstrap 增强（模板填充+门禁注入）+ finalize | B2, B19, B34 |
| Phase 6 | 策略选择验证 + DECIDE 历史过滤 | B11 |

| 维度 | 当前 (R11) | 预期 | 原因 |
|------|-----------|------|------|
| 度量效度 | 5.5 | 7.5+ | VERIFY 自动评分，TSV 自动追加，无人工遗漏 |
| 闭环性 | 5.7 | 8.0+ | 经验库自动读写，轮间状态自动传递 |
| 收敛性能 | 6.0 | 8.0+ | 振荡检测+停滞检测+max_rounds 强制 |
| 门禁判别力 | 6.1 | 7.5+ | Hard Gate 自动暂停，side_effect 自动验证 |
| 任务适配度 | 6.3 | 7.5+ | 模板特定门禁自动注入，bootstrap 完整 |
| 自进化 | 5.6 | 8.0+ | 经验库功能化，策略历史自动消费 |
| **加权总分** | **5.8** | **7.5-8.0** | |
