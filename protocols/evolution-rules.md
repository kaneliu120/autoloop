# Evolution Rules — 轮间进化规则

## 概述

AutoLoop 的核心能力是在迭代过程中动态调整目标、范围和策略。本文档定义什么时候应该调整，如何调整，以及如何记录调整。

**核心原则**：进化是受控的，不是无限蔓延。每次调整都必须有明确的触发条件和边界。

**统一状态枚举**：状态枚举定义见 `protocols/loop-protocol.md` 统一状态枚举章节，所有进化记录中必须使用该章节定义的枚举值，不得使用其他状态词（如"完成"、"未完成"、"待定"）。

---

## 进化类型 1：扩展范围

### 触发条件
- Subagent 发现了计划外的重要维度，且这个维度对目标有显著影响
- 判断标准：如果忽略这个维度，结论会有实质性偏差

### 扩展规则

**允许扩展**：
```
新维度重要性 = 高（影响核心结论）
+ 预算剩余 ≥ 30%
+ 扩展后总维度 ≤ 初始维度 × 1.5
→ 允许扩展，下一轮覆盖新维度
```

**不允许扩展**：
```
预算剩余 < 30%
OR 扩展会导致总维度翻倍
OR 新维度与核心目标关系模糊
→ 记录新维度到 findings.md "拓展方向"区块，不纳入本次范围
```

### 扩展操作步骤
1. 在 `autoloop-plan.md` 的"范围定义 > 扩展维度"中追加新维度
2. 在"变更记录"中记录：时间、新增维度、触发原因
3. 更新覆盖率计算的分母（总维度数增加）
4. 在 `autoloop-progress.md` 当前轮的 EVOLVE 部分记录决策

## 进化类型 2：收窄焦点

### 触发条件
- 已消耗 ≥ 70% 预算，但覆盖率 < 60%
- 连续 2 轮在 ≥ 3 个维度同时落后
- 发现某些维度信息极度稀缺，继续投入收益极低

### 收窄规则

**收窄优先级**：保留对核心结论影响最大的维度，降低或删除次要维度。

**哪些维度可以降低要求**：
```
维度重要性 = 低（对推荐/结论影响 < 10%）
→ 将质量标准从"有 3 个来源"降低到"有 1 个来源即可"
→ 在报告中注明"该维度调研深度有限"
```

**哪些维度可以完全跳过**：
```
维度重要性 = 低 AND 信息获取极困难 AND 预算告急
→ 从覆盖率计算中移除该维度（不计入分母）
→ 在报告中明确声明"未调研此维度及原因"
```

### 收窄操作步骤
1. 在 `autoloop-plan.md` 的"范围变更"中记录缩减内容和原因
2. 更新质量门禁（调低的维度注明调整）
3. 在 `autoloop-progress.md` 记录决策和影响评估

## 进化类型 3：策略切换

### 触发条件
- 同一维度连续 2 轮改进 < 当前分数的 3%（相对值）
  - 示例：当前 80%，改善阈值 = 2.4%；当前 7/10 分，改善阈值 = 0.21 分
- 当前方法在该维度已到极限

### 策略切换规则

记录已尝试方法，选择未尝试的方向：

**知识类任务策略矩阵**：

| 已尝试 | 切换到 |
|--------|--------|
| 关键词搜索 | 直接访问官方文档 |
| 英文搜索 | 中文搜索（针对中国市场数据）|
| 一般搜索 | 学术/专业数据库 |
| 近期内容 | 历史内容（了解演变轨迹）|
| 文字来源 | 视频/播客（技术大会演讲）|

**工程类任务策略矩阵**：

| 已尝试 | 切换到 |
|--------|--------|
| 算法优化 | 架构重构 |
| 查询优化 | 缓存策略 |
| 代码审查发现问题 | 运行 profiler 找实际瓶颈 |
| 逐行修复 | 批量替换（正则/脚本）|

### 策略切换操作步骤
1. 在 `autoloop-findings.md` 的第 2 层（策略复盘）表格中，将已失效策略标记为 `避免`，新策略标记为 `待验证`
2. 在 `autoloop-plan.md` 的"策略历史"中追加（格式见下方"进化决策日志格式"）
3. 在 ORIENT 阶段明确说明新策略，引用 findings.md 中的 `避免` 策略，避免重复

## 进化类型 4：优先级重排

### 触发条件
- 发现紧急 P1 问题（数据丢失风险、生产安全漏洞）
- 发现高于预期的关键风险

### 紧急 P1 问题判断标准

> 严重级别枚举（P1/P2/P3）定义见 `protocols/loop-protocol.md` 统一状态枚举章节。以下是触发"立即升级"处理流程的 P1 子类。

工程类：
```
紧急 P1（立即停止一切，优先处理）：
- SQL 注入已被利用的证据
- 密钥/密码暴露在代码仓库
- 生产数据被误删的风险
- 内存泄漏导致服务崩溃的模式
```

知识类：
```
紧急 P1（立即通知用户）：
- 调研目标在法律/合规上有重大风险
- 发现核心假设完全错误（如市场方向判断错误）
```

### 紧急 P1 处理流程
1. 立即暂停当前轮次的其他工作
2. 通知用户（暂停等待确认，等待用户回复）
3. 获得确认后，紧急 P1 问题的修复优先于所有其他任务
4. 修复完成后继续正常迭代

### 优先级重排的记录
```markdown
### 优先级变更记录
- 时间：{时间}
- 触发原因：{紧急 P1 问题描述}
- 暂停的任务：{列表}
- 新优先级顺序：{紧急P1修复 → 其他P1 → P2 → ...}
- 恢复时间（预计）：{估计}
```

## 进化约束（防止无限蔓延）

### 范围扩展上限
- 最多扩展 2 次（整个任务周期内）
- 每次扩展后的总维度 ≤ 初始维度 × 1.5
- 累计扩展维度数 ≤ 初始维度数的 50%

### 预算追加规则
- AutoLoop 不自动追加预算
- 达到最大轮次后，必须用户明确批准才能追加
- 追加时向用户说明：当前状态、距离目标的差距、预计还需几轮

### 目标漂移防护
- 不允许在迭代过程中更改核心目标（只能在 plan.md 的"范围定义"中追加，不能修改核心目标）
- 如果发现最初目标定义有问题，必须暂停并让用户重新确认目标
- 目标确认后，之前的迭代结果是否有效需要评估

## 进化决策日志格式

每次进化决策在 `autoloop-progress.md` 的当前轮次末尾追加：

```markdown
### 进化决策（第 N 轮末）

触发条件：{满足了哪条进化规则}

决策类型：{扩展范围 / 收窄焦点 / 策略切换 / 优先级重排}

决策内容：
- 变更前：{现有状态}
- 变更后：{调整后状态}

影响评估：
- 对覆盖率的影响：{+/-X% 或 不变}
- 对预算的影响：{需要额外 N 轮 / 节省 N 轮}
- 对最终质量的预期影响：{提升 / 降低（原因）}

策略状态更新（统一枚举）：
| 策略 | 状态 | 原因 |
|------|------|------|
| {策略A} | 避免 | {失效原因} |
| {策略B} | 待验证 | {切换到此策略的理由} |
| {策略C} | 保持 | {继续有效} |

遗留问题状态（统一枚举）：
| 问题 ID | 描述 | 状态 |
|---------|------|------|
| {ID1} | {描述} | 跨轮遗留 |
| {ID2} | {描述} | 待处理 |

下一轮策略：{具体行动计划}
```

**状态枚举使用规则**：

- `保持`：该策略上一轮有效，继续使用
- `避免`：该策略已尝试 ≥ 2 轮无改善，不再重复
- `待验证`：新引入的策略，尚未有足够数据
- `新发现`：本轮新识别的问题，尚未开始处理
- `已修复`：问题已在本轮或历史轮修复并验证
- `待处理`：已识别但未排入本轮计划
- `跨轮遗留`：上一轮标记为"待处理"，但本轮仍未完成

## Protocol 进化规则

Protocol 文件不是静态的。以下规则定义了如何通过 REFLECT 机制更新 protocol。

### 触发条件

在 REFLECT 阶段（第4层：经验教训），如果发现以下情况，应提出 protocol 更新建议：
1. 现有门禁规则导致误判（漏检或误报）
2. TSV schema 无法表达某类数据
3. 参数词汇表缺少新变量
4. 状态枚举无法表达某种状态
5. 验证方法（L1/L2）在实践中发现局限

### 更新流程

1. 在 REFLECT 的经验教训中记录具体问题和建议修改
2. 下一轮 OBSERVE 读取 REFLECT 时，评估建议的合理性
3. 如果合理，在当前轮次中更新对应的 protocol 文件
4. 更新后在 progress 中记录变更（如"更新 quality-gates.md T6 复合判定规则"）

### 约束

- Protocol 更新必须通过 REFLECT→OBSERVE 闭环，不允许在执行过程中临时修改
- 更新后所有引用该 protocol 的文件自动生效（因为是运行时读取，不是复制）
- 参数词汇表变更必须同步更新激活矩阵
