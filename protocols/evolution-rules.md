# Evolution Rules — 轮间进化规则

## 概述

AutoLoop 的核心能力是在迭代过程中动态调整目标、范围和策略。本文档定义什么时候应该调整，如何调整，以及如何记录调整。

**核心原则**：进化是受控的，不是无限蔓延。每次调整都必须有明确的触发条件和边界。

**统一状态枚举**：状态枚举定义见 `protocols/loop-protocol.md` 统一状态枚举章节，所有进化记录中必须使用该章节定义的枚举值，不得使用其他状态词（如"完成"、"未完成"、"待定"）。

---

## 进化类型 1：扩展范围

### 触发条件
- Subagent 发现了计划外的重要维度，且这个维度对目标有显著影响
- 判断标准：如果忽略这个维度，结论会有实质性偏差

### 扩展规则

**允许扩展**：
```
新维度重要性 = 高（影响核心结论）
+ 预算剩余 ≥ 30%
+ 扩展后总维度 ≤ 初始维度 × 1.5
→ 允许扩展，下一轮覆盖新维度
```

**不允许扩展**：
```
预算剩余 < 30%
OR 扩展会导致总维度翻倍
OR 新维度与核心目标关系模糊
→ 记录新维度到 findings.md "拓展方向"区块，不纳入本次范围
```

### 扩展操作步骤
1. 在 `autoloop-plan.md` 的"范围定义 > 扩展维度"中追加新维度
2. 在"变更记录"中记录：时间、新增维度、触发原因
3. 更新覆盖率计算的分母（总维度数增加）
4. 在 `autoloop-progress.md` 当前轮的 EVOLVE 部分记录决策

---

## 进化类型 2：收窄焦点

### 触发条件
- 已消耗 ≥ 70% 预算，但覆盖率 < 60%
- 连续 2 轮在 ≥ 3 个维度同时落后
- 发现某些维度信息极度稀缺，继续投入收益极低

### 收窄规则

**收窄优先级**：保留对核心结论影响最大的维度，降低或删除次要维度。

**哪些维度可以降低要求**：
```
维度重要性 = 低（对推荐/结论影响 < 10%）
→ 将质量标准从"有 3 个来源"降低到"有 1 个来源即可"
→ 在报告中注明"该维度调研深度有限"
```

**哪些维度可以完全跳过**：
```
维度重要性 = 低 AND 信息获取极困难 AND 预算告急
→ 从覆盖率计算中移除该维度（不计入分母）
→ 在报告中明确声明"未调研此维度及原因"
```

### 收窄操作步骤
1. 在 `autoloop-plan.md` 的"范围变更"中记录缩减内容和原因
2. 更新质量门禁（调低的维度注明调整）
3. 在 `autoloop-progress.md` 记录决策和影响评估

---

## 进化类型 3：策略切换

### 触发条件
- 同一维度连续 2 轮改进 < 当前分数的 3%（相对值）
  - 示例：当前 80%，改善阈值 = 2.4%；当前 7/10 分，改善阈值 = 0.21 分
- 当前方法在该维度已到极限

### 策略切换规则

记录已尝试方法，选择未尝试的方向：

**知识类任务策略矩阵**：

| 已尝试 | 切换到 |
|--------|--------|
| 关键词搜索 | 直接访问官方文档 |
| 英文搜索 | 中文搜索（针对中国市场数据）|
| 一般搜索 | 学术/专业数据库 |
| 近期内容 | 历史内容（了解演变轨迹）|
| 文字来源 | 视频/播客（技术大会演讲）|

**工程类任务策略矩阵**：

| 已尝试 | 切换到 |
|--------|--------|
| 算法优化 | 架构重构 |
| 查询优化 | 缓存策略 |
| 代码审查发现问题 | 运行 profiler 找实际瓶颈 |
| 逐行修复 | 批量替换（正则/脚本）|

### 策略切换操作步骤
1. 在 `autoloop-findings.md` 的第 2 层（策略复盘）表格中，将已失效策略标记为 `避免`，新策略标记为 `待验证`
2. 在 `autoloop-plan.md` 的"策略历史"中追加（格式见下方"进化决策日志格式"）
3. 在 ORIENT 阶段明确说明新策略，引用 findings.md 中的 `避免` 策略，避免重复

## 进化类型 4：优先级重排

### 触发条件
- 发现紧急 P1 问题（数据丢失风险、生产安全漏洞）
- 发现高于预期的关键风险

### 紧急 P1 问题判断标准

工程类：
```
紧急 P1（立即停止一切，优先处理）：
- SQL 注入已被利用的证据
- 密钥/密码暴露在代码仓库
- 生产数据被误删的风险
- 内存泄漏导致服务崩溃的模式
```

知识类：
```
紧急 P1（立即通知用户）：
- 调研目标在法律/合规上有重大风险
- 发现核心假设完全错误（如市场方向判断错误）
```

### 紧急 P1 处理流程
1. 立即暂停当前轮次的其他工作
2. 通知用户（人工确认点，等待确认）
3. 获得确认后，紧急 P1 问题的修复优先于所有其他任务
4. 紧急 P1 修复完成后继续正常迭代

### 优先级重排的记录
```markdown
### 优先级变更记录
- 时间：{时间}
- 触发原因：{紧急 P1 问题描述}
- 暂停的任务：{列表}
- 新优先级顺序：{紧急P1修复 → 其他P1 → ...}
- 恢复时间（预计）：{估计}
```

---

## 进化约束（防止无限蔓延）

### 范围扩展上限
- 最多扩展 2 次（整个任务周期内）
- 每次扩展后的总维度 ≤ 初始维度 × 1.5
- 累计扩展维度数 ≤ 初始维度数的 50%

### 预算追加规则
- AutoLoop 不自动追加预算
- 达到最大轮次后，必须用户明确批准才能追加
- 追加时向用户说明：当前状态、距离目标的差距、预计还需几轮

### 目标漂移防护
- 不允许在迭代过程中更改核心目标（只能在 plan.md 的"范围定义"中追加，不能修改核心目标）
- 如果发现最初目标定义有问题，必须暂停并让用户重新确认目标
- 目标确认后，之前的迭代结果是否有效需要评估

## 协议进化流程（Protocol Evolution）

本章节定义 AutoLoop 协议文件本身发生变更时的标准流程。协议进化与轮间进化（范围/焦点/策略调整）不同：前者修改的是制度层文件（`protocols/` 目录），后者调整的是当次任务的执行方式。

**核心原则**：协议是唯一规则源，修改必须受控、可追溯、向下兼容。

```
触发 → 提案 → 审批 → 更新 → 通知 → 验证 → 冲突处理 → 存档
```

### 步骤 1：触发

**谁执行**：REFLECT 阶段中的 subagent（通常是 code-reviewer 或 verifier）。

**输入**：当轮迭代发现的问题——现有协议规则不合理、存在歧义、缺少覆盖场景，或与实际运行产生矛盾。

**输出**：一条变更需求，记录到当轮 `autoloop-progress.md` 的 REFLECT 部分，格式：
```
协议变更需求：
- 文件：{protocols/xxx.md}
- 问题：{具体描述，引用触发该问题的场景}
- 影响范围：{哪些流程受影响}
```

**记录位置**：`autoloop-progress.md` 当前轮 REFLECT 区块。

---

### 步骤 2：提案

**谁执行**：当轮编排 agent（负责该轮 REFLECT 阶段的主 agent）。

**输入**：步骤 1 的变更需求。

**输出**：完整变更提案，包含：
- 改什么（具体章节、当前内容、建议内容）
- 为什么（根因分析，不是表面描述）
- 影响范围（哪些 commands / templates 需要联动修改）
- 不改的理由（评估过但排除的替代方案）

**记录位置**：`autoloop-progress.md` 当前轮 REFLECT 区块，紧接变更需求之后。

---

### 步骤 3：审批

**谁执行**：用户（Kane）。

**输入**：步骤 2 的完整变更提案。

**输出**：确认、否决或修改意见。

**默认规则**：用户明确确认后方可执行更新。用户可否决全部或要求修改提案内容；否决的提案记录原因后存档，不执行更新。

### 变更风险分级与审批规则

| 变更类型 | 风险等级 | 审批方式 | 示例 |
|---------|---------|---------|------|
| 新增评分锚点/校准样本 | 低 | AI自动推进 | 为T4补充7分样本描述 |
| 补充经验条目到策略效果库 | 低 | AI自动推进 | 记录新策略的效果数据 |
| 修正校准样本 | 低 | AI自动推进 | 根据实际数据调整锚点 |
| 调整参数（±20%以内） | 中 | AI推进 + 下轮验证 | 默认轮次从5改为4 |
| 新增/删除评分维度 | 高 | 用户确认 | 为T1增加"实用性"维度 |
| 修改门禁阈值 | 高 | 用户确认 | T6安全性从9改为8 |
| 修改状态机/流程 | 高 | 用户确认 | 增加新阶段 |
| 删除现有规则 | 高 | 用户确认 | 移除某个扣分项 |

**规则**：低风险变更 AI 直接执行并在 progress.md 中记录；中风险变更 AI 执行但下轮 OBSERVE 必须验证效果，无效则自动回滚；高风险变更必须暂停等待用户确认后执行。

### 协议升级后重基线规则

minor 或 major 版本变更后，下一个任务的首轮 OBSERVE 必须用新标准重新评分基线。

**重基线流程**：

1. 检测 `protocol_version` 是否与上次任务不同
2. 如果版本变更（minor/major），对同一对象用新标准重新评分所有维度
3. 记录重基线结果到 `experience-registry.md` 变更日志：
   - `protocol_version`: {old} → {new}
   - 各维度：旧分（旧标准）→ 新分（新标准）
   - 分差 = 标准变更的影响量（非质量变化）
4. 以新基线作为后续迭代的起点

**豁免**：patch 版本变更（锚点/样本补充）不需要重基线，因为不改变评分标准本身。

**目的**：确保协议自进化的投入产出比可评估，避免新标准下的分数回退被误判为质量下降。

**记录位置**：`autoloop-progress.md` 当前轮 REFLECT 区块，记录审批结果（确认 / 否决 / 修改后确认）及用户意见。

---

### 步骤 4：更新

**谁执行**：当轮编排 agent。

**输入**：经用户确认的变更提案。

**输出**：修改后的 protocol 文件（精确修改，不影响无关内容）；如提案涉及联动修改，同步更新对应 commands / templates 文件，但不引入协议层之外的额外改动。

**操作要求**：
- 精确修改指定章节，不改动无关内容
- 修改完成后立即执行语法和引用一致性自检

**记录位置**：`autoloop-progress.md` 当前轮，记录已修改的文件列表和具体章节。

---

### 步骤 5：通知

**谁执行**：当轮编排 agent，在更新完成后立即执行。

**输入**：步骤 4 的修改结果。

**输出**：变更通知，写入 `autoloop-progress.md` 当前轮，格式：
```
协议变更通知：
- 文件：{protocols/xxx.md}
- 章节：{具体章节名}
- 变更内容：{一句话描述变更}
- 生效轮次：{从下一轮起生效}
- 联动修改：{涉及的 commands / templates 文件，若无则填"无"}
```

**记录位置**：`autoloop-progress.md` 当前轮，置于「协议变更通知」标题下，与步骤 3 审批记录同块。

---

### 步骤 6：验证

**谁执行**：下一轮迭代启动时的 ORIENT 阶段 subagent。

**输入**：步骤 5 的变更通知（从 `autoloop-progress.md` 读取）。

**输出**：验证报告，确认：
1. 新规则已在本轮执行中生效
2. 联动修改的文件内容与协议一致
3. 未发现与其他规则的冲突

**记录位置**：`autoloop-progress.md` 下一轮 ORIENT 区块，附「协议验证」子标题。

**验证阶段2：结果验证（新增）**

合规验证确认"改动已写入"，结果验证确认"改动有效果"。

1. **预期目标声明**：每次协议变更必须在提案阶段声明预期目标
   - 格式：预期{指标}从{当前值}改善到{目标值}，验证窗口{N}次任务
   - 示例："预期T6评分方差从±1.5降低到±0.8，验证窗口3次任务"
   - 每次变更幅度控制：预期改善 ≤ 20%，渐进式推进，不做大跃进

2. **效果追踪**：在验证窗口内的每次任务中，记录与预期目标相关的实际指标

3. **结果判定**：
   - 达到预期 → 固化为正式规则，protocol_version patch+1
   - 未达预期但有正向趋势 → 延长验证窗口1次
   - 未达预期且无正向趋势 → 触发回滚评估（不自动回滚，评估后决定）

4. **记录**：结果记录到 `experience-registry.md` 协议变更效果追踪表

**追踪表使用规范**：

协议变更效果追踪表（位于 `experience-registry.md`）的每一行对应一次协议变更，必须在以下时机更新：

| 时机 | 操作 | 更新字段 |
|------|------|---------|
| 提案通过时 | 创建新行 | change_id, protocol_version, 变更内容, 预期目标, 验证窗口 |
| 验证窗口内每次任务结束 | 追加实际数据 | 实际效果（追加本次指标值） |
| 验证窗口结束时 | 判定结果 | 状态（已达标/未达标） |
| 回滚执行后 | 记录回滚 | 状态（已回滚），实际效果（追加回滚原因） |

**回滚评估具体判据**：

```text
触发回滚评估的条件（非自动回滚，需评估后决定）：
1. 验证窗口内 0 次正向改善
2. 任一相关维度出现 ≥ 1.0 分回退
3. 验证窗口结束时未达预期目标且无正向趋势

回滚评估流程：
1. 分析失败原因：变更本身有问题 vs 执行环境不同 vs 样本不足
2. 如果变更有问题 → 回滚变更，protocol_version patch-1
3. 如果环境不同 → 保留变更但标注适用条件
4. 如果样本不足 → 延长验证窗口 1 次（仅允许延长 1 次）
```

---

### 冲突处理

当新协议规则与现有规则产生矛盾时：

1. **新规则优先**：经用户确认的新规则具有最高效力，覆盖旧规则
2. **冲突项识别**：在步骤 6 验证时识别，记录到 `autoloop-progress.md` 当前轮「冲突清单」
3. **冲突解决排期**：冲突项自动排入下一轮 REFLECT 讨论，生成新的变更需求（回到步骤 1）
4. **临时处理**：在冲突解决前，执行 agent 以最新变更通知为准，旧规则中与之矛盾的部分暂时失效

---

### 存档

- 所有协议变更记录（变更需求 → 提案 → 审批 → 通知 → 验证）永久保留在 `autoloop-progress.md` 对应轮次，不可删除
- 存档内容作为后续 REFLECT 阶段的历史参考，防止重复提出已否决的变更
- 若任务结束后需要汇总协议变更历史，从各轮 `autoloop-progress.md` 的「协议变更通知」区块提取

## 进化决策日志格式

每次进化决策在 `autoloop-progress.md` 的当前轮次末尾追加：

```markdown
### 进化决策（第 N 轮末）

触发条件：{满足了哪条进化规则}

决策类型：{扩展范围 / 收窄焦点 / 策略切换 / 优先级重排}

决策内容：
- 变更前：{现有状态}
- 变更后：{调整后状态}

影响评估：
- 对覆盖率的影响：{+/-X% 或 不变}
- 对预算的影响：{需要额外 N 轮 / 节省 N 轮}
- 对最终质量的预期影响：{提升 / 降低（原因）}

策略状态更新（统一枚举）：
| 策略 | 状态 | 原因 |
|------|------|------|
| {策略A} | 避免 | {失效原因} |
| {策略B} | 待验证 | {切换到此策略的理由} |
| {策略C} | 保持 | {继续有效} |

遗留问题状态（统一枚举）：
| 问题 ID | 描述 | 状态 |
|---------|------|------|
| {ID1} | {描述} | 跨轮遗留 |
| {ID2} | {描述} | 待处理 |

下一轮策略：{具体行动计划}
```

**状态枚举使用规则**：

- `保持`：该策略上一轮有效，继续使用
- `避免`：该策略已尝试 ≥ 2 轮无改善，不再重复
- `待验证`：新引入的策略，尚未有足够数据
- `新发现`：本轮新识别的问题，尚未开始处理
- `已修复`：问题已在本轮或历史轮修复并验证
- `待处理`：已识别但未排入本轮计划
- `跨轮遗留`：上一轮标记为"待处理"，但本轮仍未完成
