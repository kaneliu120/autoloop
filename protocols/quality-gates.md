# Quality Gates — 质量门禁规范

## 概述

质量门禁是 AutoLoop 迭代的**首要终止标准（成功路径）**。本文档定义所有任务类型的质量门禁，包括计算方法、评分规则和通过标准。

**终止层级**：
1. **质量门禁全部达标** → 成功终止（优先路径）
2. **用户中断** → 暂停，保存进度，可恢复
3. **预算耗尽（达到最大轮次）** → 输出当前最优结果，明确标注未达标项
4. **阻塞（无法继续推进）** → 上报用户，说明阻塞原因和所需信息

质量门禁是成功终止的判断标准；预算、中断、阻塞是其他终止路径，均在 `protocols/loop-protocol.md` 中有完整处理流程。

**核心原则**：质量门禁必须是**数字化的、可测量的**，不接受"差不多好了"这样的描述。

## 知识类任务门禁（T1/T2/T3/T4）

### 覆盖率（Coverage）

**定义**：已经有实质内容的维度数 / 计划调研的总维度数

**计算方法**：

```text
已覆盖维度 = 在 findings.md 中有至少 2 个具体信息点（不是"未找到"）的维度数
覆盖率 = 已覆盖维度 / 总维度数 × 100%
```

注：阈值定为 2 个信息点（而非 3 个），是因为信息点数量反映覆盖广度，信息的可信度由独立的"可信度"门禁负责（≥ 2 个独立来源）。两者分工不重叠。

**通过标准**：
- T1 Research：≥ 85%
- T2 Compare：100%（所有选项的所有维度都必须有内容）
- T4 Generate：每个生成单元都完成 = 100%

**示例计算**：

```text
计划维度：8 个
有实质内容（≥2个信息点）：功能(3个来源) + 性能(4个来源) + 成本(2个来源) + 社区(5个来源) + 成熟度(2个来源) = 5个
信息不足（<2个信息点）：学习曲线(1个来源) + 文档(0个来源) + 风险(1个来源) = 3个
覆盖率 = 5/8 = 62.5%（未达标）
```

---

### 可信度（Credibility）

**定义**：有 ≥ 2 个独立来源支撑的关键发现 / 总关键发现数

**什么是"独立来源"**：
- 同一作者不同文章 → 不算独立
- 同一公司不同产品页 → 不算独立
- 同一论坛不同帖子 → 不算独立（除非作者不同且有原创内容）
- 官方文档 + 第三方评测 → 算独立
- 不同媒体报道 + 官方数据 → 算独立

**信息来源可信度分级**：

```text
级别 1（高可信）：官方文档/官方 GitHub/一手研究报告
级别 2（中高）：Gartner/Forrester/IDC/Stack Overflow/Hacker News 高票
级别 3（中）：知名技术媒体、专家博客（有实名认证）
级别 4（低中）：一般博客、Reddit 普通回答
级别 5（低）：匿名内容、未注明时间的内容
```

**计算方法**：

```text
关键发现 = 在 findings.md 中明确标注为"关键结论"的陈述
有多源支撑 = 该陈述有 ≥2 个独立来源，且至少一个是级别 1-2
可信度 = 有多源支撑的关键发现 / 总关键发现 × 100%
```

**通过标准**：≥ 80%

---

### 一致性（Consistency）

**定义**：无矛盾维度数 / 总维度数 × 100%

注：以**维度**为计量单位（而非单条陈述）。维度内部有任一矛盾陈述，该维度即计为矛盾维度。

**什么算矛盾**：
- 同一属性的不同数据（A 说性能 100ms，B 说 500ms，且无解释）
- 逻辑矛盾（优势和劣势中说了同一件事）
- 时间范围不明确导致的明显差异

**什么不算矛盾**（需要在 findings 中说明）：
- 不同使用场景下的不同表现
- 不同时间版本的不同结果
- 主观评价的差异

**计算方法**：

```text
矛盾维度 = 在 cross-verification 报告中标注为"存在矛盾"的维度数
无矛盾维度 = 总维度数 - 矛盾维度数
一致性 = (无矛盾维度 / 总维度) × 100%
```

**通过标准**：≥ 90%

---

### 完整性（Completeness）

**定义**：有引用来源的关键陈述数 / 总关键陈述数

**关键陈述 vs 一般陈述**：
- 关键陈述：用于支撑结论或推荐的陈述（必须有来源）
- 一般陈述：背景描述、解释性内容（建议有来源但不强制）

**计算方法**：

```text
关键陈述 = 在 findings.md 中的结论类、数据类、推荐依据类陈述
有来源 = 陈述后有 (来源: URL) 或 [^N] 引用标注
完整性 = 有来源的关键陈述 / 总关键陈述 × 100%
```

**通过标准**：≥ 85%

## T2 专属门禁

### 偏见检查（Bias Check）

**定义**：每个候选选项由 ≥ 2 个独立 subagent 评估，且各 subagent 对同一选项的评分差异在 0-10 量表上 < 1.5 分。

**唯一计算公式（所有文件以此为准）**：

```text
对每个选项：
  偏见分数 = (max_score - min_score) / 10
  通过条件：偏见分数 < 0.15（即 10 分量表上最大差异 < 1.5 分）
  不通过：触发第 3 个独立 analyzer 重新评估该选项

偏见检查通过 = 所有选项均满足上述条件
```

**计算示例**：

```text
选项 A：analyzer_1 = 7.5，analyzer_2 = 8.2
  偏见分数 = (8.2 - 7.5) / 10 = 0.07 → 通过

选项 B：analyzer_1 = 6.0，analyzer_2 = 8.5
  偏见分数 = (8.5 - 6.0) / 10 = 0.25 → 不通过，触发第 3 个独立 analyzer
```

**通过标准**：所有选项的偏见分数 < 0.15（差异 ≥ 1.5 分须运行第三次独立评估，以多数结论为准）

---

### 敏感性分析（Sensitivity Analysis）

**定义**：将 plan 中 `key_assumptions` 列表中的每个关键假设分别调整 ±20%，检查最终推荐排名第 1 位是否保持不变。

`key_assumptions` 格式（来自 `autoloop-plan.md`，结构化列表，每项含 name + current_value + unit）：

```text
key_assumptions:
  - name: 实施周期
    current_value: 3
    unit: 月
  - name: 月度预算上限
    current_value: 5000
    unit: USD
```

**唯一计算公式（所有文件以此为准）**：

```text
对 plan.key_assumptions 中每个假设 H：
  original_rank = 用 H.current_value 计算的推荐排名
  high_rank = 用 H.current_value × 1.2 计算的排名
  low_rank  = 用 H.current_value × 0.8 计算的排名
  通过条件：original_rank == high_rank == low_rank（第 1 位不变）

敏感性分析通过 = 所有 key_assumptions 均满足上述条件
```

**通过标准**：任意单一关键假设 ±20% 变动后，推荐排名第 1 位不变

## T3 专属门禁

### KPI 目标（KPI Target）

**定义**：用户在 autoloop-plan.md 中定义的 KPI 数值达到 plan 中设定的目标阈值。

**计算方法**：

```text
测量值 = VERIFY 阶段对 KPI 指标的实际测量结果
通过 = 测量值 达到或优于 plan.md 中设定的目标阈值
```

**要求**：plan 必须明确写出 KPI 名称、测量方法和目标阈值（例："API 响应时间 < 200ms"、"覆盖率 ≥ 90%"）。当 KPI 定义缺失时，在 OBSERVE 阶段阻塞（进入 AWAIT_USER 状态），要求用户提供 KPI 定义、目标值和测量方法后继续。

## T4 专属门禁

### 通过率（Pass Rate）

**定义**：通过质检的内容数 / 总生成数 × 100%

**计算方法**：

```text
通过质检 = 满足 plan 中定义的内容标准（非空、格式正确、无明显错误）的内容项数量
通过率 = 通过质检的内容数 / 总生成数 × 100%
```

**通过标准**：≥ 95%

### 平均分（Average Score）

**定义**：所有内容项质量评分的算术平均值（评分标准须在 autoloop-plan.md 中定义，范围 0-10）

**计算方法**：

```text
各内容项得分由 subagent 按 plan 中定义的评分标准打分（0-10）
平均分 = sum(所有内容项得分) / 总内容项数
```

**通过标准**：≥ 7/10

## 工程类任务门禁（T5/T6/T7）

### 语法验证门禁（强制，阶段 1 结束前）

```bash
# 使用 plan 中定义的 syntax_check_cmd 验证所有修改文件
# syntax_check_file_arg=true:  {syntax_check_cmd} {文件路径}（逐文件验证）
# syntax_check_file_arg=false: {syntax_check_cmd}（项目级验证，不附加文件名）

# 通过标准：零错误（有 warning 可以接受，但必须记录）
```

### 验证层级说明

AutoLoop 的验证分为两个层级：

| 层级 | 名称 | 方法 | 可信度 | 适用场景 |
|------|------|------|--------|---------|
| L1 | 近似检查 | grep/文本搜索 | 中（可能有假阳性：匹配到注释、import、未使用变量） | 快速验证，用于门禁初筛 |
| L2 | 精确验证 | 框架特定pattern/AST分析/运行时测试 | 高 | 最终验收，用于关键门禁 |

门禁中标注 `[L1]` 或 `[L2]` 表示该验证的层级。当 L1 检查通过但执行者仍不确信时，应升级到 L2。

### 路由注册门禁（当 new_router_name ≠ N/A 时）

**[L1] 近似检查**（快速初筛）：
按技术栈选择对应 grep pattern：
- Python/FastAPI: `grep -n "include_router.*{new_router_name}" {main_entry_file}`
- Node.js/Express: `grep -n "use.*{new_router_name}\|router.*{new_router_name}" {main_entry_file}`
- 其他: `grep -n "{new_router_name}" {main_entry_file}`

通过标准：grep 找到匹配行。

**已知局限**：L1 检查可能将 import 语句、注释、未使用变量误判为注册。如果 grep 结果中没有明确的注册/挂载语句，需人工确认或升级到 L2。

**[L2] 精确验证**（可选，推荐用于关键门禁）：
- 人工审查 grep 结果，确认匹配行是实际的注册/挂载语句（不是 import 或注释）
- 或运行项目的启动命令，确认路由可访问

## T5 N/A 处理规则

```text
如果 service_list 为空或 N/A：Phase 4 服务检查标记为 N/A（跳过）
如果 health_check_url 为空：Phase 4 健康检查标记为 N/A（跳过）
对于 script/library 类型的 project_type，两者均可为 N/A（无常驻服务）
Phase 5：始终需要用户输入 'verified'，不受 N/A 影响
```

## T6 复合判定规则（唯一规则，所有文件以此为准）

T6 达标条件为复合判定，以下两个条件**必须同时满足**：

**条件一：分数达标**

```text
安全性 ≥ 9/10
可靠性 ≥ 8/10
可维护性 ≥ 8/10
```

**条件二：计数达标**

```text
P1 问题 = 0（所有维度）
安全 P2 问题 = 0
可靠性 P2 问题 ≤ 3
可维护性 P2 问题 ≤ 5
```

**计数优先原则**：即使分数已达标，P1 问题 > 0 则不达标，必须继续修复 P1 后才能终止。

> 本规则是 T6 达标的唯一判定标准。`delivery-phases.md` Phase 2 中 P2=0 是 T5 的交付标准，不适用于 T6。两者不冲突：T5 要求交付代码零 P2，T6 允许遗留少量低风险 P2 问题。

## 门禁豁免规则

在以下情况可以记录豁免（不是跳过，是明确说明为什么可以接受）：

1. **遗留代码豁免**：不在本次修改范围内的遗留问题，可以记录但不强制修复
2. **技术债豁免**：P3 级别的问题，在预算耗尽时可以记录到"遗留问题"，不阻塞交付
3. **环境限制豁免**：线上验收因环境问题无法执行的步骤，需要说明替代验证方法

**豁免必须**：
- 明确说明豁免原因
- 评估影响（是否影响核心功能）
- 记录到 progress.md 和最终报告的"遗留问题"部分
- 不得隐藏（不是"忘了"，是"知道了，决定不修复，原因是..."）
