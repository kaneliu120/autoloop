# Quality Gates — 质量门禁规范

## 概述

质量门禁是 AutoLoop 迭代的**首要终止标准（成功路径）**。本文档定义所有任务类型的质量门禁，包括计算方法、评分规则和通过标准。

**终止层级**：
1. **质量门禁全部达标** → 成功终止（优先路径）
2. **用户中断** → 暂停，保存进度，可恢复
3. **预算耗尽（达到最大轮次）** → 输出当前最优结果，明确标注未达标项
4. **无法继续推进** → 上报用户，说明原因和所需信息

质量门禁是成功终止的判断标准；预算、中断、无法继续是其他终止路径，均在 `protocols/loop-protocol.md` 中有完整处理流程。

**核心原则**：质量门禁必须是**数字化的、可测量的**，不接受"差不多好了"这样的描述。

## 评分语义词汇表（所有文件以此为准）

AutoLoop 使用 4 种不同语义的评分概念，不可混用：

| 概念 | 英文标识 | 类型 | 取值范围 | 含义 | 使用场景 |
|------|---------|------|---------|------|---------|
| 质量得分 | quality_score | 连续值 | 0-10（保留1位小数） | 维度的当前质量水平 | TSV的metric_value列；门禁评估矩阵的阈值比较 |
| 置信度 | confidence | 百分比 | 0-100%（整数） | 评分本身的可信程度 | TSV的confidence列；fail-closed判定 |
| 严重度 | severity | 枚举 | P1 / P2 / P3 | 发现问题的影响程度 | enterprise-standard.md扣分规则；门禁计数条件 |
| 门禁状态 | gate_status | 枚举 | 通过 / 未通过 / 豁免 | 门禁的最终判定 | 门禁评估矩阵；终止条件判断 |

**混用禁止规则**：
- quality_score 不可用于表示置信度（"这个评分质量高"应该用 confidence）
- severity 不可用于替代 quality_score（P1/P2/P3 是问题分类，不是分数）
- confidence 低 ≠ quality_score 低（可能质量高但证据不足）
- gate_status 由 quality_score + 阈值比较 + 计数条件 共同决定，不可独立赋值

**跨文件一致性**：所有 AutoLoop 文件在使用这 4 个概念时，必须使用上表中的英文标识或中文名称，不得使用同义词（如不可用"评分"模糊指代 quality_score 和 confidence 两种不同概念）。

## 评分通用规范（所有模板适用）

**证据强制要求**：所有评分必须同时输出分数、判据、证据三项，缺少任一项的评分无效。

```text
评分输出格式：
- 分数：{数字}
- 判据：命中了{区间}的{具体条目}
- 证据：{文件路径+行号} 或 {来源URL+引用段落}
无证据的评分视为未通过（fail-closed）。该维度必须在下轮补充证据后重新评分。缺证据不是"跳过"，而是强制系统反思、发现线索和问题后才能推进。
```

**评分区间通用锚点**：

| 区间 | 含义 | 判据要求 |
|------|------|---------|
| 1-3分 | 明确缺陷 | 列出缺什么、错什么 |
| 4-6分 | 部分满足 | 列出有什么、还缺什么 |
| 7-8分 | 基本满足 | 列出满足了哪些标准 |
| 9-10分 | 优秀 | 列出超出预期的具体方面 |

**评分分歧仲裁协议**（唯一仲裁规则，所有模板适用）：

```text
仲裁触发：同一维度多个 evaluator 的 quality_score 差异 ≥ 2.0 分
仲裁流程：
  1. 取较低分作为当前轮次的 quality_score
  2. 在 TSV 的 score_variance 列记录方差值
  3. 在 TSV 的 details 列标注 "分歧仲裁：{evaluator_1}={分数1}, {evaluator_2}={分数2}"
  4. 下轮 OBSERVE 必须将该分歧列为优先处理项
  5. 如有第 3 个独立 evaluator，以 2/3 多数意见为准

不触发仲裁的情况：
  - 差异 < 2.0 分 → 取均值，无需仲裁
  - 单 evaluator → 方差 = 0，无需仲裁
```

**评分置信度计算**：

```text
置信度 = f(证据数量, evaluator一致性)
- 证据 ≥ 3条 + 方差 < 1.0 → 高置信（≥ 80%）
- 证据 1-2条 + 方差 < 2.0 → 中置信（50-79%）
- 证据 0条 或 方差 ≥ 2.0 → 低置信（< 50%）→ fail-closed（视为未通过）
```

低置信评分与缺证据评分同等处理：视为未通过，必须在下轮补充证据或增加evaluator后重新评分。

### 跨模型评分一致性机制

当使用多个 LLM（如 Claude + Codex）作为 evaluator 时，采用以下机制确保评分一致性：

**结构化 rubric 优先**：所有评分必须基于本文档的评分锚点表（而非自由提示），确保不同模型使用相同的评分标准。evaluator 的 prompt 中必须包含对应模板的锚点表全文。

**异构 panel 评估**：
- T6/T7 等高风险模板：建议 ≥ 2 个不同模型独立评分
- 各 evaluator 独立产出分数+判据+证据，互不可见（盲评）
- 评分差异处理：见上方"评分分歧仲裁协议"

**晚聚合（Late Fusion）**：
- 各 evaluator 先独立完成全部维度评分
- 全部完成后统一聚合（取均值或仲裁），不在评分过程中交换信息
- 聚合时机：VERIFY 阶段结束、写入 TSV 之前

**交换顺序去偏（Swap Augmentation）**：
- T2 Compare 中，不同 evaluator 以不同顺序评估选项（evaluator_1 从 A→B→C，evaluator_2 从 C→B→A）
- 消除顺序效应导致的评分偏差
- 其他模板：如评分维度 ≥ 5 个，建议 evaluator 以不同顺序评估维度

## 知识类任务门禁（T1/T2/T3/T4）

### 覆盖率（Coverage）

**定义**：已经有实质内容的维度数 / 计划调研的总维度数

**计算方法**：

```text
已覆盖维度 = 在 findings.md 中有至少 2 个具体信息点（不是"未找到"）的维度数
覆盖率 = 已覆盖维度 / 总维度数 × 100%
```

注：阈值定为 2 个信息点（而非 3 个），是因为信息点数量反映覆盖广度，信息的可信度由独立的"可信度"门禁负责（≥ 2 个独立来源）。两者分工不重叠。

**通过标准**：
- T1 Research：≥ 85%
- T2 Compare：100%（所有选项的所有维度都必须有内容）
- T4 Generate：每个生成单元都完成 = 100%

**示例计算**：

```text
计划维度：8 个
有实质内容（≥2个信息点）：功能(3个来源) + 性能(4个来源) + 成本(2个来源) + 社区(5个来源) + 成熟度(2个来源) = 5个
信息不足（<2个信息点）：学习曲线(1个来源) + 文档(0个来源) + 风险(1个来源) = 3个
覆盖率 = 5/8 = 62.5%（未达标）
```

**覆盖率评分锚点**：

| 区间 | 判据 |
|------|------|
| 1-3 | <50% 维度有实质内容 |
| 4-6 | 50-84% 有实质内容 |
| 7-8 | 85-95% 有实质内容 |
| 9-10 | >95% 且每维度 ≥ 3 信息点 |

---

### 可信度（Credibility）

**定义**：有 ≥ 2 个独立来源支撑的关键发现 / 总关键发现数

**什么是"独立来源"**：
- 同一作者不同文章 → 不算独立
- 同一公司不同产品页 → 不算独立
- 同一论坛不同帖子 → 不算独立（除非作者不同且有原创内容）
- 官方文档 + 第三方评测 → 算独立
- 不同媒体报道 + 官方数据 → 算独立

**信息来源可信度分级**：

```text
级别 1（高可信）：官方文档/官方 GitHub/一手研究报告
级别 2（中高）：Gartner/Forrester/IDC/Stack Overflow/Hacker News 高票
级别 3（中）：知名技术媒体、专家博客（有实名认证）
级别 4（低中）：一般博客、Reddit 普通回答
级别 5（低）：匿名内容、未注明时间的内容
```

**计算方法**：

```text
关键发现 = 在 findings.md 中明确标注为"关键结论"的陈述
有多源支撑 = 该陈述有 ≥2 个独立来源，且至少一个是级别 1-2
可信度 = 有多源支撑的关键发现 / 总关键发现 × 100%
```

**通过标准**：≥ 80%

**可信度评分锚点**：

| 区间 | 判据 |
|------|------|
| 1-3 | <50% 关键发现有独立来源 |
| 4-6 | 50-79% 有独立来源 |
| 7-8 | ≥ 80% 有独立来源 |
| 9-10 | >90% 且关键发现 ≥ 3 独立来源 |

---

### 一致性（Consistency）

**定义**：无矛盾维度数 / 总维度数 × 100%

注：以**维度**为计量单位（而非单条陈述）。维度内部有任一矛盾陈述，该维度即计为矛盾维度。

**什么算矛盾**：
- 同一属性的不同数据（A 说性能 100ms，B 说 500ms，且无解释）
- 逻辑矛盾（优势和劣势中说了同一件事）
- 时间范围不明确导致的明显差异

**什么不算矛盾**（需要在 findings 中说明）：
- 不同使用场景下的不同表现
- 不同时间版本的不同结果
- 主观评价的差异

**计算方法**：

```text
矛盾维度 = 在 cross-verification 报告中标注为"存在矛盾"的维度数
无矛盾维度 = 总维度数 - 矛盾维度数
一致性 = (无矛盾维度 / 总维度) × 100%
```

**通过标准**：≥ 90%

**一致性评分锚点**：

| 区间 | 判据 |
|------|------|
| 1-3 | 存在未记录的矛盾信息 |
| 4-6 | 矛盾已记录但未解决 |
| 7-8 | 所有矛盾已解决并有说明 |
| 9-10 | 无矛盾或全部解决有交叉验证证据 |

---

### 完整性（Completeness）

**定义**：有引用来源的关键陈述数 / 总关键陈述数

**关键陈述 vs 一般陈述**：
- 关键陈述：用于支撑结论或推荐的陈述（必须有来源）
- 一般陈述：背景描述、解释性内容（建议有来源但不强制）

**计算方法**：

```text
关键陈述 = 在 findings.md 中的结论类、数据类、推荐依据类陈述
有来源 = 陈述后有 (来源: URL) 或 [^N] 引用标注
完整性 = 有来源的关键陈述 / 总关键陈述 × 100%
```

**通过标准**：≥ 85%

## T2 专属门禁

### 偏见检查（Bias Check）

**定义**：每个候选选项由 ≥ 2 个独立 subagent 评估，且各 subagent 对同一选项的评分差异在 0-10 量表上 < 1.5 分。

**唯一计算公式（所有文件以此为准）**：

```text
对每个选项：
  偏见分数 = (max_score - min_score) / 10
  通过条件：偏见分数 < 0.15（即 10 分量表上最大差异 < 1.5 分）
  不通过：触发第 3 个独立 option-analyzer 重新评估该选项

偏见检查通过 = 所有选项均满足上述条件
```

**计算示例**：

```text
选项 A：option-analyzer_1 = 7.5，option-analyzer_2 = 8.2
  偏见分数 = (8.2 - 7.5) / 10 = 0.07 → 通过

选项 B：option-analyzer_1 = 6.0，option-analyzer_2 = 8.5
  偏见分数 = (8.5 - 6.0) / 10 = 0.25 → 不通过，触发第 3 个独立 option-analyzer
```

**通过标准**：所有选项的偏见分数 < 0.15（差异 ≥ 1.5 分须运行第三次独立评估，以多数结论为准）

### 敏感性分析（Sensitivity Analysis）

**定义**：将 plan 中 `key_assumptions` 列表中的每个关键假设分别调整 ±20%，检查最终推荐排名第 1 位是否保持不变。

`key_assumptions` 格式（来自 `autoloop-plan.md`，结构化列表，每项含 name + current_value + unit）：

```text
key_assumptions:
  - name: 实施周期
    current_value: 3
    unit: 月
  - name: 月度预算上限
    current_value: 5000
    unit: USD
```

**唯一计算公式（所有文件以此为准）**：

```text
对 plan.key_assumptions 中每个假设 H：
  original_rank = 用 H.current_value 计算的推荐排名
  high_rank = 用 H.current_value × 1.2 计算的排名
  low_rank  = 用 H.current_value × 0.8 计算的排名
  通过条件：original_rank == high_rank == low_rank（第 1 位不变）

敏感性分析通过 = 所有 key_assumptions 均满足上述条件
```

**通过标准**：任意单一关键假设 ±20% 变动后，推荐排名第 1 位不变

## T3 专属门禁

### KPI 目标（KPI Target）

**定义**：用户在 autoloop-plan.md 中定义的 KPI 数值达到 plan 中设定的目标阈值。

**计算方法**：

```text
测量值 = VERIFY 阶段对 KPI 指标的实际测量结果
通过 = 测量值 达到或优于 plan.md 中设定的目标阈值
```

**要求**：plan 必须明确写出 KPI 名称、测量方法和目标阈值（例："API 响应时间 < 200ms"、"覆盖率 ≥ 90%"）。当 KPI 定义缺失时，在 OBSERVE 阶段暂停（进入暂停等待确认状态），要求用户提供 KPI 定义、目标值和测量方法后继续。

## T4 专属门禁

### 通过率（Pass Rate）

**定义**：通过质检的内容数 / 总生成数 × 100%

**计算方法**：

```text
通过质检 = 满足 plan 中定义的内容标准（非空、格式正确、无明显错误）的内容项数量
通过率 = 通过质检的内容数 / 总生成数 × 100%
```

**通过标准**：≥ 95%

### 平均分（Average Score）

**定义**：所有内容项质量评分的算术平均值（评分标准须在 autoloop-plan.md 中定义，范围 0-10）

**计算方法**：

```text
各内容项得分由 subagent 按 plan 中定义的评分标准打分（0-10）
平均分 = sum(所有内容项得分) / 总内容项数
```

**通过标准**：≥ 7/10

## T6 复合判定规则（唯一规则，所有文件以此为准）

## 硬门禁 vs 软门禁分类（所有文件以此为准）

门禁分为硬门禁（必须通过）和软门禁（建议通过），不允许混合解释。具体门禁项以上方"门禁评估矩阵"为唯一来源，本章节仅标注硬/软属性。

### 分类规则

**硬门禁（Hard Gate）**：失败 = 整轮不通过，必须在下轮修复后才能继续。
**软门禁（Soft Gate）**：失败 = 记录到 progress.md + findings.md 但可继续迭代，不阻塞终止。

| 模板 | 硬门禁 | 软门禁 |
|------|--------|--------|
| T1 | 覆盖率、可信度 | 一致性、完整性 |
| T2 | 覆盖率、偏见检查 | 敏感性分析 |
| T3 | KPI 达目标值 | — |
| T4 | 通过率、平均分 | — |
| T5 | 语法验证、P1/P2=0、人工验收 | 服务健康检查（N/A 可跳过） |
| T6 | 安全性 ≥ 9/10、可靠性 ≥ 8/10、可维护性 ≥ 8/10、P1=0、安全 P2=0 | 可靠性 P2 ≤ 3、可维护性 P2 ≤ 5 |
| T7 | 架构 ≥ 8/10、性能 ≥ 8/10、稳定性 ≥ 8/10 | — |

**T6 特别说明**：T6 的**分数阈值**（安全 ≥ 9、可靠 ≥ 8、可维护 ≥ 8）和 **P1=0 + 安全P2=0** 是硬门禁；可靠性/可维护性的 **P2 计数容忍**（≤ 3/≤ 5）是软门禁。两者不矛盾：分数必须达标，但允许少量低风险 P2 遗留。完整复合判定见上方"T6 复合判定规则"。

**判定规则**：
- 所有硬门禁通过 + 软门禁全部记录 → gate_status = 通过
- 任一硬门禁未通过 → gate_status = 未通过（不论软门禁状态）
- 软门禁未通过但硬门禁全部通过 → gate_status = 通过（软门禁问题记录为遗留项）

## 门禁豁免规则

在以下情况可以记录豁免（不是跳过，是明确说明为什么可以接受）：

1. **遗留代码豁免**：不在本次修改范围内的遗留问题，可以记录但不强制修复
2. **技术债豁免**：P3 级别的问题，在预算耗尽时可以记录到"遗留问题"，不影响交付
3. **环境限制豁免**：线上验收因环境问题无法执行的步骤，需要说明替代验证方法

**豁免必须**：
- 明确说明豁免原因
- 评估影响（是否影响核心功能）
- 记录到 progress.md 和最终报告的"遗留问题"部分
- 不得隐藏（不是"忘了"，是"知道了，决定不修复，原因是..."）
